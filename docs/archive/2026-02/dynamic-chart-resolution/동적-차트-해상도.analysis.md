#### H-01: Zoom Threshold Logic Difference (HIGH)

**Design** (Section 3.2 + Plan Section 8.1):
```
Level 0: 50%~100% -> 15m
Level 1: 10%~50% -> 1m
Level 2: 2%~10%  -> 10s
Level 3: <2%     -> 1s
```

The design implies a direct ratio-to-level mapping (i.e., if ratio drops below 50%, jump to 1m; below 10%, jump to 10s).

**Implementation** (`chart-utils.ts:40-57`):
```typescript
const ZOOM_IN_THRESHOLD = 0.50;  // zoom in when < 50%
const ZOOM_OUT_THRESHOLD = 0.75; // zoom out when > 75%
```

The implementation uses a **step-by-step** approach: it only moves one level at a time per zoom event. For example, at 15m with ratio=5%, the design implies jumping directly to 10s (Level 2), but the implementation only steps to 1m (Level 1). This requires **multiple successive zoom operations** to reach deeper levels.

Additionally, the zoom-out threshold (75%) is not mentioned in the design at all -- it is an implementation-only addition.

**Impact**: HIGH -- User must perform multiple zoom gestures to reach Level 2/3 when starting from Level 0, which differs from the design's implied direct mapping. However, this may be an intentional UX improvement (preventing jarring jumps from 15m directly to 10s).

#### H-02: Mock Data Field Name Mismatch (HIGH)

**Design** API response (Section 4.2):
```json
{ "time": "08:00:00", "power": 3.4 }
```

**Implementation** (range-data.ts:50-54):
```typescript
data.push({
  time,
  current,  // NOT "power" or "air"
  prev,     // NOT in design at all
});
```

The mock data uses `current`/`prev` instead of `power`/`air` as defined in the design's API response. When the real Backend API returns `power`/`air`, the chart series configuration (which maps to `current`/`prev`) will break.

The TrendChart series in MON002LineDetail is configured with `key: 'current'` and `key: 'prev'`, which matches the mock but not the design's API spec.

**Impact**: HIGH -- Backend integration will require either:
1. Changing the Backend response format to use `current`/`prev`, or
2. Adding a transformation layer in `fetchRangeData`, or
3. Updating the chart series config and mock data to use `power`/`air`/`prevPower`/`prevAir`.

#### M-04: Air Chart Does Not Independently Trigger Zoom (MEDIUM)

**Design** (Section 2.1, 5.1) implies both power and air charts independently respond to zoom:
```
TrendChart (전력)  ... onZoomChange(zoomRatio) 콜백 호출
TrendChart (에어)  ... (동일 구조)
```

**Implementation** (MON002LineDetail.tsx:267):
```tsx
onZoomChange={undefined}  // Air chart does NOT trigger zoom
```

Only the power chart triggers dynamic resolution. The air chart follows the power chart's interval via `useEffect` sync (line 83-87). This means:
- Zooming on the air chart does NOT trigger interval change
- Only power chart zoom triggers resolution changes for both charts

This is likely intentional (avoiding conflicting zoom levels between two charts) but deviates from the design's implied equal behavior.

#### L-03: Toggle Switch Not in Design (LOW)

The implementation adds a checkbox toggle "동적 해상도 (Progressive Resolution)" that is not part of the design. This allows users to switch between legacy (fixed interval) and dynamic resolution modes. This is a pragmatic addition for backward compatibility but should be documented in the design.

## 9. Quality Assessment

### 9.1 Code Quality

| Aspect | Score | Notes |
|--------|:-----:|-------|
| Type Safety | 95% | All functions properly typed; only `[key: string]: any` in DataPoint |
| Error Handling | 70% | Infinite loop prevention excellent; API error feedback lacks user notification |
| Performance | 90% | SWR caching, debouncing, downsampling all implemented; zoom lock prevents excessive calls |
| Readability | 90% | Well-commented JSDoc, clear variable names, logical structure |
| Maintainability | 85% | Clean separation of concerns; hook pattern is reusable |

### 9.2 Strengths

1. **Infinite Loop Prevention**: Three layers of protection (isZoomingRef, intervalChangedRef, zoomLockRef) effectively prevent zoom-API-render cycles
2. **SWR Caching**: Interval-specific cache keys enable level-independent caching as designed
3. **Backward Compatibility**: Toggle between legacy and dynamic modes preserves existing functionality
4. **Debouncing Strategy**: Double debouncing (500ms in hook + 700ms in TrendChart) prevents API flooding
5. **ResizeObserver Integration**: TrendChart properly resizes with container (fixes previous migration gap)
6. **Type Completeness**: All custom types (Interval, ZoomLevel, RangeDataResponse, DynamicResolutionOptions) are well-defined

### 9.3 Concerns

1. **Zero Tests**: No unit, integration, or E2E tests exist for this feature (or the entire project)
2. **Mock Data Shape Mismatch**: `current`/`prev` fields will not match backend's `power`/`air` fields
3. **Single-step Zoom**: Users may find it frustrating to zoom 3 times to reach 1-second data from 15-minute view
4. **No Performance Monitoring**: Design mentions `performance.mark/measure` but not implemented

### 13.1 H-01 수정: 한 단계씩 이동 + Zoom Out 제거 (2026-02-28)

**이슈**: Zoom ratio에 따른 다단계 점프 지원 (Design 3.2 vs Implementation)

**원인 분석**:
- Gap Analysis 시 Design 문서의 "Level 0: 50%~100%", "Level 1: 10%~50%" 표를 다단계 점프로 잘못 해석
- 실제 Design Section 2.2 Data Flow는 모든 예시가 한 단계씩 이동 (50%→30%: 15m→1m, 30%→5%: 1m→10s)
- 사용자 요구사항: "한번에 무조건 한단계씩", "2단계 건너뛰기 방지", "역순서 안 됨"

**수정 내용** (`apps/web/src/lib/chart-utils.ts:44-55`):

```typescript
// 수정 전 (다단계 점프 로직)
if (zoomRatio < 0.10) {
  targetIndex = 3; // 0→3 직접 점프 가능 (3단계 건너뜀)
} else if (zoomRatio < 0.25) {
  targetIndex = 2; // 0→2 직접 점프 가능 (2단계 건너뜀)
} else if (zoomRatio < 0.50) {
  targetIndex = 1;
} else if (zoomRatio > 0.75 && currentIndex > maxLevel) {
  targetIndex = currentIndex - 1; // Zoom out 자동 전환
}

// 수정 후 (한 단계씩 + zoom in만)
if (zoomRatio < 0.50 && currentIndex < levels.length - 1) {
  targetIndex = currentIndex + 1; // 무조건 +1만 (한 단계씩)
}
// Zoom out 자동 전환 제거 (리셋 버튼으로만 복귀)
else {
  targetIndex = currentIndex;
}
```

**효과**:
- ✅ 15m → 1m → 10s → 1s 순서 보장 (한 번에 1단계씩만)
- ✅ 2단계 이상 건너뛰기 완전 차단
- ✅ 역방향 자동 전환 제거 (1s → 10s → 1m → 15m 불가)
- ✅ 리셋 버튼으로만 초기 interval 복귀

**Design 문서 업데이트 필요**:
- [ ] Section 3.2: 한 단계씩 zoom 동작 명시
